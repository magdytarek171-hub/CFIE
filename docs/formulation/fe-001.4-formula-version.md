# FE-001.4: Formula Version

## Purpose

`Formula` objects are frozen and carry a single caller-supplied `version` label with no
system-enforced meaning or history. There is currently no way to record that formula
content changed, in what order, or by whom, without discarding the prior state.

`FormulaVersion` closes that gap. It is an immutable record that captures one complete
`Formula` snapshot at a point in time, together with the minimal metadata required to
order snapshots into a lineage. It fulfills the versioning intent already stated in
`docs/formulation/formula-data-model.md`: edits never mutate a `Formula`; they produce a
new snapshot, and prior snapshots remain intact and reconstructable.

## Scope

In scope for this specification:

- The `FormulaVersion` data model: its fields, types, and immutability guarantees.
- The relationship between a `FormulaVersion` record and the `Formula` snapshot it wraps.
- The versioning philosophy that governs how snapshots relate to one another.
- The serialization contract that a future serializer extension must satisfy.
- The validation boundaries that a future validator extension must enforce.

Out of scope: anything described in "Explicit non-goals" below. This specification
defines a data shape, not behavior.

## Data model

`FormulaVersion` is a single immutable record. It does not replace `Formula`; it wraps
one. Each `FormulaVersion` instance contains exactly one complete, unmodified `Formula`
object as its `formula_snapshot` — never a partial formula, a diff, or a reference that
requires external lookup to resolve.

Relationship to existing models:

```text
FormulaVersion → Formula (formula_snapshot) → Phase → IngredientEntry → Ingredient UUID
```

A lineage of versions for the same formula is modeled as a set of `FormulaVersion`
records that share the same `formula_id` and are chained through
`predecessor_version_id`. The chain is linear for this specification — see
"Explicit non-goals."

## Field definitions

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `version_id` | `UUID` | Yes | Stable identifier for this specific version record. Unique per record, never reused. |
| `formula_id` | `UUID` | Yes | Identifies the formula lineage this version belongs to. Must equal `formula_snapshot.formula_id`. |
| `version_number` | `int` | Yes | Positive integer position of this record within its lineage (`1` for the first recorded version). System-assigned; distinct from the caller-supplied `Formula.version` free-text label. |
| `formula_snapshot` | `Formula` | Yes | The complete, immutable `Formula` object captured at this version. No partial or delta content. Named explicitly (rather than `snapshot`) to leave room for future sibling fields such as `ingredient_snapshot` or `process_snapshot`. |
| `recorded_at` | `datetime` | Yes | Timezone-aware timestamp when this version record was created. Distinct from `formula_snapshot.created_at` / `formula_snapshot.modified_at`, which describe the formula content itself. |
| `recorded_by` | `str` | Yes | Caller-supplied identifier of who produced this version record. May differ from `formula_snapshot.author`. |
| `change_summary` | `str \| None` | No | Optional caller-supplied description of what changed relative to the predecessor. Defaults to `None`. |
| `predecessor_version_id` | `UUID \| None` | No | `version_id` of the immediately preceding version in this lineage. `None` only for the first version. Defaults to `None`. |

## Immutability rules

- `FormulaVersion` is a `@dataclass(frozen=True, slots=True)`, matching every existing
  formulation model.
- The wrapped `formula_snapshot` is itself a frozen `Formula`; `FormulaVersion` adds no
  field that allows the snapshot's content to be altered after construction.
- There is no setter, mutator, or in-place update method. Any change in formula state is
  represented by constructing a new `FormulaVersion` instance with a new `version_id`
  and an incremented `version_number` — never by modifying an existing record.
- Two `FormulaVersion` records are equal only if every field, including the nested
  `formula_snapshot`, is equal — consistent with standard dataclass value semantics.
- Collections referenced transitively through `formula_snapshot` (phases, ingredients,
  notes) remain tuples, per the existing `Formula` and `Phase` models.

## Versioning philosophy

A `Formula` is never edited in place. Producing a new state of a formula means:

1. Constructing a new, independent `Formula` object with the same `formula_id` and the
   updated content.
2. Wrapping it in a new `FormulaVersion` record with an incremented `version_number` and
   a `predecessor_version_id` pointing at the prior version.

This makes formula history a plain, append-only sequence of immutable records. No
existing `FormulaVersion` or `Formula` object is ever changed or deleted to produce
history — the full lineage remains reconstructable at any time by following
`predecessor_version_id` links back to the version with `predecessor_version_id = None`.

`version_number` and `Formula.version` serve different purposes and are not required to
match: `version_number` is a system-assigned, monotonically increasing sequence within a
lineage. Gapless enforcement requires visibility across multiple records and belongs to
a future version management service; `Formula.version` remains a free-text label a
caller may set to anything (e.g. `"Draft A"`, `"2.1-pilot"`).

Versioning in this specification is strictly linear per lineage — one predecessor, one
successor. Branching and merging are explicitly deferred (see "Explicit non-goals").

## Serialization requirements

`FormulaVersion` has no serializer implementation yet; this section defines the contract
a future serializer extension must satisfy, consistent with the existing
`formula_to_dict` / `formula_from_dict` / `to_json` / `from_json` / `to_yaml` /
`from_yaml` functions in `backend/core/formulation/serializer.py`:

- The nested `formula_snapshot` must be converted using the existing `formula_to_dict` /
  `formula_from_dict` functions unchanged — no duplicate or divergent formula
  serialization logic.
- `version_id` and `predecessor_version_id` serialize as strings (`str(UUID)` /
  `UUID(str)`); `predecessor_version_id` serializes as `null` / `None` when absent.
- `recorded_at` serializes via `.isoformat()` and deserializes via
  `datetime.fromisoformat()`, matching `created_at` / `modified_at` handling on
  `Formula`.
- `version_number` serializes as a plain JSON/YAML integer.
- `change_summary` serializes as a string or `null` / `None`.
- JSON serialization uses the standard library `json` module only.
- YAML serialization uses only PyYAML's `safe_dump()` and `safe_load()`.
- Serialization performs no persistence, network, or filesystem I/O.

## Validation boundaries

`FormulaVersion` carries no validation logic itself, consistent with every other
formulation model. This section defines the boundary conditions a future validator
extension (parallel to `validate_formula` in `validator.py`) must enforce; it does not
implement them:

- `version_number` must be a positive integer (`>= 1`).
- `formula_id` must equal `formula_snapshot.formula_id`.
- `predecessor_version_id`, when present, must not equal `version_id` (a version cannot
  be its own predecessor).
- `formula_snapshot` must independently satisfy `validate_formula`; `FormulaVersion`
  does not re-implement or relax formula-level structural rules.
- `recorded_at` must be timezone-aware, matching the existing constraint implied for
  `Formula.created_at` / `modified_at`.

Enforcing sequential numbering across an entire lineage (no gaps, no duplicates,
correct predecessor chain) requires visibility across multiple records and is explicitly
a future validator/service concern, not a constraint this data model enforces on a
single instance.

## Explicit non-goals

- No validator implementation. This ticket defines boundaries only.
- No serializer implementation. This ticket defines the contract only.
- No automatic `version_number` assignment, sequencing, or persistence logic.
- No branching or merging support — a version has at most one predecessor and this
  specification does not model multiple successors.
- No diff, delta, or comparison computation between two `Formula` snapshots.
- No rollback, revert, or "restore to version" behavior.
- No storage, database, or repository adapter of any kind.
- No API or service-layer exposure.
- No approval workflow, permissions, or business rules about who may record a version.
- No changes to the existing `Formula`, `Phase`, `IngredientEntry`, or enum models.
